# Runiq Context Travel Guide

Runiq Context Travel Guide demonstrates how a .NET application can attach domain-specific source documents to an agent and let the agent produce grounded travel-planning answers from retrieved context.

The sample shows a clear separation of responsibilities: context search provides destination knowledge, tools provide supporting signals, and the agent creates the itinerary and final response.

## Why This Sample Exists

LLMs are useful for planning and summarization, but application-specific answers should be grounded in the documents and services that the host application provides.

The agent does not need to know every destination upfront. The host application gives it searchable context sources, deterministic tools, and Runiq makes the retrieval and tool execution visible in the dashboard.

## What This Sample Demonstrates

- Registering agents from a dedicated `Agents` folder
- Attaching source documents through a Runiq context space
- Searching context before answering travel questions
- Showing searched documents and selected excerpts in the dashboard
- Using typed .NET tools for deterministic weather and budget support
- Keeping itinerary generation in the agent, not in a tool
- Keeping `Program.cs` focused on application wiring

## Agent

The sample registers two agents:

- `travel-agent`, displayed as `Travel Agent`
- `plain-agent`, displayed as `Plain Agent`

`Travel Agent` is the main sample agent. It is responsible for practical city trip planning and is attached to the `travel-planning` context space. It can use:

- travel source documents under `Context`
- the `weather` tool
- the `travel_budget_estimate` tool
- the trip-planning skill under `Skills/trip-planning/SKILL.md`

The itinerary is generated by the agent. The tools only provide supporting signals such as weather and approximate budget.

`Plain Agent` is a simple assistant without tools or context. It is included as a contrast to the context-aware travel agent.

## Context And Source Documents

Context files are domain-specific knowledge sources provided by the host application. In this sample, the context space contains travel guides for Istanbul, Ankara, Bursa, and Izmir, plus a PDF travel/history source.

The agent can search these files before answering. The dashboard shows whether context search happened, how many documents were searched, and whether a relevant excerpt was selected.

Context search is not the same thing as a tool call. Context search retrieves relevant text from attached documents. A tool call executes application code.

## Tools

### WeatherTool

`weather` returns deterministic sample weather information for a city. It does not call a real weather service. The Travel Agent uses it when the user asks for weather-aware planning, weather, or current conditions.

### TravelBudgetEstimateTool

`travel_budget_estimate` returns a deterministic demo budget estimate based on:

- city
- group size
- day count
- travel style: `budget`, `standard`, or `premium`

The budget tool does not read context and does not create a trip plan. It only estimates a simple demo budget from city, group size, day count, and travel style.

### ServerTimeTool

`server_time` returns the current server UTC time. It is registered as a standalone tool for the dashboard tool playground, not attached to the Travel Agent.

## Expected Dashboard Behavior

For a plan plus budget prompt:

- A Context searched card should appear.
- `weather` may appear if weather or current conditions are requested.
- `travel_budget_estimate` should appear for budget or cost requests.
- The final answer should include the itinerary first.
- The final answer should include the approximate budget estimate at the end.

For a budget-only prompt:

- `travel_budget_estimate` should run.
- No itinerary is required.

For an itinerary-only prompt:

- Context search should run.
- `travel_budget_estimate` should not be required unless the user asks for budget or cost.

For a question outside the provided context:

- Context search may show no selected excerpt or no strong match.
- The agent should be transparent that the attached context does not contain enough information.

Tool cards appear separately from context cards.

## Deterministic Budget Example

For this prompt:

```text
15 kişilik bir grup için İzmir'de 2 günlük tarihi gezi planı hazırla ve en alta yaklaşık standart bütçe ekle.
```

The standard estimate uses these per-person, per-day values:

- meal: 1,000 TRY
- local transport: 400 TRY
- guide/activity: 400 TRY
- total per person per day: 1,800 TRY

Calculation:

```text
15 people * 2 days * 1,800 TRY = 54,000 TRY
```

Expected budget output:

- Estimated meal cost: 30,000 TRY
- Estimated local transport cost: 12,000 TRY
- Estimated guide/activity cost: 12,000 TRY
- Estimated total cost: 54,000 TRY

Accommodation, intercity transportation, museum tickets, and personal expenses are not included.

## Context Search Vs Tool Calls

Context search:

- retrieves relevant text from documents
- does not execute business logic
- is useful for guides, manuals, knowledge base documents, policies, and travel notes

Tool calls:

- execute typed .NET code
- can query databases, APIs, files, or deterministic services
- are useful for structured data, calculations, transactions, and workflow actions

`Runiq.AI.ExpenseDesk` is a good example of typed tool access to structured data. `Runiq.AI.ContextTravelGuide` is a good example of context-grounded answering.

## Running The Sample

From the repository root:

```bash
cd samples/Runiq.AI.ContextTravelGuide
dotnet run
```

Open the dashboard at:

```text
http://localhost:6185/dashboard
```

If ASP.NET Core prints a different port, use the printed port.

## Configuration

The sample uses OpenAI through the existing Runiq agent provider setup. Configure the API key through `appsettings.Development.json`, user secrets, or environment variables.

Example `appsettings.Development.json` shape:

```json
{
  "OpenAI": {
    "ApiKey": "YOUR_OPENAI_API_KEY"
  }
}
```

Do not commit `appsettings.Development.json`. Use the Development environment for local debugging.

`Properties/launchSettings.json` sets `ASPNETCORE_ENVIRONMENT=Development` and contains no secrets.

## Current Scope

This is a sample application, not a production travel product. The source documents are demo assets, and the budget estimate is a deterministic demo calculation. The purpose is to demonstrate context retrieval, source-grounded answering, tool visibility, and dashboard inspection.
